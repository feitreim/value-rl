# Correctness Testing

To ensure our custom model implementation (`model.py`) and training logic (`grpo.py`, `gwen.py`) are numerically correct and differentiable, we use `test_correctness.py`.

## Differentiability
- **batch_logprobs**: Verified that gradients flow through the custom Metal kernels when calculating logprobs.
- **GRPO Loss**: Verified that the full GRPO loss function is differentiable.

## Numerical Parity with vllm-mlx / mlx-lm
We compare our model's output to the standard `mlx-lm` implementation (which `vllm-mlx` uses).

### Logprobs
- We compare the logprobs extracted by `batch_logprobs` with those computed manually using `mlx_lm.load()`.
- **Finding**: There is a small numerical difference (max diff ~0.375 in BF16). This is likely due to differences in RoPE or RMSNorm implementations between our custom Metal kernels and MLX's `fast` ops, or subtle differences in how the chat template is handled.
- **Threshold**: We currently accept a max difference of 0.5 for logprobs in BF16.

### Greedy Sampling
- We compare the tokens generated by our `chat` function (greedy mode, `temp=0.0`) with those generated by `vllm-mlx`'s `EngineCore`.
- **Finding**: Greedy tokens match exactly for the first 10 tokens on a simple prompt ("What is 2+2?"). This confirms that the forward pass is functionally equivalent to the reference implementation.

## Batch Consistency & Fused MLP Kernel

### Root Cause of Batch-Size-Dependent Divergence
BF16 matmuls use different GPU tiling strategies depending on batch size, producing tiny rounding
differences that compound through 28 residual transformer layers. Investigation traced the sources:
- **MLP matmuls**: bf16 gate/up/down projections amplify input diffs ~16x (0.002 → 0.016).
  fp32 accumulation reduces this to 0.14x (0.002 → 0.0003).
- **Attention `o_proj`**: the 2048-dim dot product introduces ~0.0005 diff per layer on identical
  inputs. This is the sole remaining source of batch-dependent divergence.
- **q/k/v_proj, RoPE, SDPA**: all produce 0.0 diff on identical inputs. Not contributors.

### Fused MLP Metal Kernel (`gwen.py`: `fused_mlp`)
A single Metal kernel computes the entire MLP `down(silu(gate(x)) * up(x))` with **fp32
accumulators**. The intermediate `h` (3072 × fp32 = 12KB) lives entirely in threadgroup shared
memory — never written to global memory or truncated to bf16.

**Kernel design** (one threadgroup per token row, 1024 threads):
1. Load input row `x` (1024 bf16) into shared memory
2. **Phase 1**: each thread computes 3 intermediate values (3072/1024) — two K=1024 fp32 dot
   products (gate + up) → silu → multiply → write to `h_shared[N]` (fp32)
3. Barrier
4. **Phase 2**: each thread computes 1 output value — N=3072 fp32 dot product reading from
   `h_shared`, writes bf16 result

**Weight layout**: transposed + combined at first use for coalesced access:
- `gate_up_wt = concat(gate_proj.weight.T, up_proj.weight.T)` → shape (K=1024, 2*N=6144)
- `down_wt = down_proj.weight.T` → shape (N=3072, D=1024)

### Results
- **Inference (fused MLP)**: B≥2 is perfectly batch-deterministic (**0.0 diff** across B=2,4,8).
  B=1 vs B≥2 still differs (~0.74 logits) due to attention `o_proj` bf16 matmul.
- **Training (standard MLP)**: Uses non-fused differentiable path. B=1 vs B=8 logprob diff ~0.18.
  Unaffected since GRPO always computes old_lps/policy_lps at the same batch size.
- `batch_logprobs` also computes log-softmax in fp32 to avoid additional accumulation noise.

### Determinism Note
Standard MLX matmuls guarantee **within-batch determinism** (all items in a single batch
produce identical outputs for identical inputs). This is sufficient for RL training where
advantages are computed within a group. Across-batch determinism (B=1 vs B=8) is not
guaranteed in BF16 due to varying GPU tiling strategies, but differences are small (~0.2 logprobs)
and do not affect training stability.

### `mx.fast.metal_kernel` API Notes
- `grid` parameter = **total thread count**, NOT threadgroup count. Number of threadgroups =
  grid / threadgroup in each dimension.
- Template params are compile-time constants (dtype, array dims).
- `ensure_row_contiguous=True` guarantees memory layout for indexing math.
- Metal kernels are **not differentiable** — MLX autograd treats them as opaque.
